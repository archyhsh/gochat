豆包模拟面

===========================================
1、对于网关的设计，你提到网关与业务逻辑解耦，请问这个网关的核心职责是什么？是基于go-zero的gateway组件自研还是二次开发？网关如何做的协议转换？网关与后端微服务的通信是同步还是异步？如果网关节点宕机，你采用什么方式保证连接不中断，信息不丢失？ 
*   职责：核心职责是 “三权分立”：安全鉴权（JWT/限流）、路由分发、协议转换（BFF）。
*   实现：基于 go-zero 的 rest 模块二次开发。网关层通过 zrpc.Client 将 HTTP 请求透传给后端的 gRPC 服务。
*   协议转换：采用 “自动映射”。网关解析 JSON Payload，封装为 Protobuf 对象，通过 gRPC 的同步调用发往后端。
*   同步与异步：查询类（如用户信息）是 同步；写操作（发信息）是 异步（投递 Kafka 后立即返回）。
*   高可用：网关本身 Stateless（无状态），前面挂 Nginx/LVS 做四层负载。节点宕机后，负载均衡器秒级剔除，客户端重连机制保障连接恢复。

2、go-zero的负载均衡默认是轮询，解释一下它的轮询机制吧。你有没有做特殊的定制化？gRPC的服务发现用的是etcd，简单介绍一下原理？服务注册与反注册分别是什么？你如何把控时机？如果某个业务节点负载过高，你的负载均衡策略如何感知并动态调整？10万在线用户的情况下，微服务集群的节点数是多少？如何评估节点的扩容阈值？
*   轮询机制（P2C）：go-zero 默认使用 P2C (Power of Two Choices)。它随机选取两个节点，比较其 当前负载（EWMA 延迟），选择更优的一个。这比单纯的轮询更能避开“木桶效应”中的慢节点。
*   服务发现（Etcd）：节点启动时在 Etcd 注册 Lease（租约）并带上节点信息。客户端 Watch 该目录，Etcd 通过长连接实时推送节点变化。
*   注册与反注册时机：注册在应用启动成功、健康检查通过后；反注册在进程收到 SIGTERM 信号时，先从 Etcd 移除，再优雅关闭连接。
*   节点规模与扩容：10万用户规模下，RPC 节点约 10-20 个。扩容阈值设为 CPU 60% 或 P99 响应超过阈值。

3、你如何统计P99指标的？prometheus+grafana是具体怎么使用的？100ms的耗时拆接下来，哪个环节占比最高？有没有遇到过p99突增的情况？根因是什么？如何通过链路追踪定位优化？
*   统计方式：通过 Prometheus 埋点，直方图（Histogram）统计。
*   占比分析：100ms 中，占比最高通常是 数据库/Redis 磁盘 I/O 或 网络 RTT。
*   P99 突增根因：通常是 “长尾效应”。原因包括 GC STW（内存分配过快）、TCP 丢包重传、或数据库慢查询。
*   定位：通过 go-zero 深度集成的 OpenTelemetry，在 Grafana 查 trace_id，看哪一步的 span 变长。

4、具体用了什么削峰策略？kafka的核心功能是削峰吗？阈值是多少？如果消息队列堆积，有没有降级策略？生产端如何感知队列堆积并触发削峰策略？ 
*   策略：消息发送采用 “异步存库，同步推送”。
*   核心功能：解耦和削峰。当瞬间 TPS 超过数据库写入极限（如 5000/s）时，Kafka 作为缓冲区。
*   降级：若队列堆积，触发 “丢弃非核心通知消息” 或 “强制客户端限流”。
*   感知：通过 Prometheus 监控 Kafka 的 Consumer Lag（消费滞后量）。

5、IM场景下的幂等性解决了什么问题？幂等校验是在网关、业务还是存储层？redis做幂等如何设置key的过期时间呢？
*   幂等解决：解决了网络重试导致的 消息重复显示。
*   层级：校验在 存储层（唯一键约束）+ 缓存层（MsgID 去重）。
*   Redis Key：idempotent:msg:msg_id，过期时间设为 24-48 小时（覆盖所有可能的重试窗口）

 6、消息到达率的统计口径是什么？如果是跨节点投递如何保证这个指标？
*   口径：(成功下发消息数 / 客户端发送成功数) * 100%。
*   跨节点保证：通过 消息回执（ACK） 和 离线补偿。如果 A 发给 B，B 在另一个节点且离线，消息存入数据库，待 B 上线后通过 Sequence 差值主动拉取。

 7、消息丢失的兜底方案是什么？重试次数和间隔如何设计？有没有出现过重试导致信息延迟过高？ 
*   方案：重试 + 定时补发 + 客户端主动同步。
*   重试设计：3 次重试，间隔分别为 1s, 2s, 4s（退避算法）。
*   副作用：重试确实会导致个别消息延迟升高，但保证了最终一致性。

8、多级缓存具体是哪几级？本地缓存用的是go-zero的cache组件对吧？缓存更新策略是什么？95的缓存命中率怎么计算？缓存穿透、缓存击穿和缓存雪崩分别如何解决？有没有遇到过缓存与数据库数据不一致？如何修复？ 
*   层级：第一级是 go-zero 内置的 collection.Cache（本地内存），第二级是分布式 Redis。
*   更新策略：先更数据库，再删缓存（Cache Aside Pattern）。
*   95% 计算：通过 Prometheus 统计 cache_hit / (cache_hit + cache_miss)。
*   三害预防：穿透用布隆过滤器/空值缓存；击穿用 singleflight（go-zero自带）；雪崩用随机过期时间。

11、如果redis集群宕机，系统怎么做降级？
*   Redis 宕机：系统回退至 纯数据库操作。虽然 P99 会飙升，但业务核心（发信息）不能断。

 12、10万用户同时发送信息，如果削峰扛不住，有没有熔断机制，如何恢复？
*   策略：当错误率达到 20% 时，触发 Breaker 开启，停止对该服务的调用，5s 后尝试进入 Half-Open 状态。

13、为什么选择go-zero而不是其他微服务框架？go-zero在IM场景下的核心优势是什么？有没有看过go-zero源码？
*   核心优势：内置了极其强大的 流量治理能力。在 IM 这种极易产生瞬间峰值的场景下，go-zero 的自适应限流和熔断器比自研更稳定。
*   源码阅读：深入研究过它的缓存更新机制（防止雪崩）和基于 p2c 的负载均衡实现。
