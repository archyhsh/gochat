# GoChat v2.0.0 第一阶段：工业级微服务架构与高性能消息引擎总结

## 1. 核心架构演进 (Architectural Evolution)
- **解耦网关与边缘服务**：彻底分离 HTTP 业务网关（Gateway）与实时通信服务（Chat RPC）。利用 Redis 实现**分布式连接寻址**，支持 Chat 节点无限水平扩展，解决了长连接维护与业务逻辑的物理隔离。
- **身份隧道全线贯通**：建立基于 **gRPC Metadata** 的身份透传体系。从 API 中间件解析 JWT 到后端 RPC 逻辑获取 `user_id`，实现了全链路身份识别，彻底终结了硬编码 TODO。

## 2. 高性能消息引擎 (Message Engine)
- **写减压方案 (Read Fan-out)**：
    - **群聊**：每条消息仅更新 `conversation` 全局表一行记录（自增 Sequence），不再循环更新成员列表，写压力从 $O(N)$ 降至 $O(1)$。
    - **私聊**：保留写扩散以支撑个性化摘要与未读数。
- **Sequence 游标分页**：全面弃用不稳定且性能随页码下降的 `offset` 分页，改为基于会话内连续递增 `sequence_id` 的游标分页，确保海量记录下的查询不重不漏、极速加载。
- **自动分表路由**：
    - **月表路由**：通过解析雪花 ID 中的时间戳，精准定位物理存储表。
    - **跨月探测算法**：实现“逐月回溯”逻辑，当单月消息不足时自动跨月取数，对前端透明。
    - **自愈能力**：模型层集成 `INSERT ... ON DUPLICATE KEY` 与“写时自动建表”逻辑，实现数据底座的自动伸缩。

## 3. 异步Resilience与领域驱动 (Event Driven)
- **削峰填谷**：利用 Kafka 异步落库。网关收到消息即刻返回 ACK，由 `Message Consumer` 匀速持久化，保护数据库不受瞬时流量冲击。
- **系统通知自动化**：`Group/Relation` 服务仅负责产生事实（事件），由 `Message` 服务消费并自动翻译为可读的系统消息（如“XX加入了群组”、“您已成为好友”），实现了完美的业务解耦。

## 4. 工业级工程实践 (Engineering Excellence)
- **配置环境驱动**：引入 `.env` 管理敏感信息，YAML 支持 `${VAR:default}` 占位符，支持 Docker 容器与 Windows 宿主机无缝连接。
- **时区鲁棒性**：全系统强制使用 **UTC** 时间，入口处嵌入 `tzdata` 数据库，解决了 Windows 环境下驱动报错的顽疾。
- **前端 MVP 闭环**：实现了支持搜索发现、好友申请处理、群组成员管理及系统消息渲染的高内聚前端界面，业务逻辑完全对齐微服务契约。
