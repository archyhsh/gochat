# GoChat v2.0.0 后续优化方案 (架构演进与面试加分项)

本文档总结了基于简历优化的后续技术路径，重点围绕 **高并发 (High Concurrency)** 与 **高可用 (High Availability)** 两大核心维度。

## 1. 核心任务清单 (Milestones)

### 1.1 全链路可观测性看板 (Observability Dashboard) - [进行中]
- **目标**：建立生产级的系统监控与故障排查体系，将后端运行状态可视化。
- **核心组件**：
  - **Prometheus**：集成 go-zero 内置指标，收集 QPS、P99 延迟、CPU/内存占用、缓存命中率等。
  - **Grafana**：构建监控面板，实时观测各微服务节点的健康度与吞吐量。
  - **Jaeger (OpenTelemetry)**：实现全链路追踪，追踪消息从网关入口、Kafka 队列到 RPC 持久化的全生命周期。
- **价值**：面试时可作为“P99 指标”的有力证据，展示对系统稳定性维护的工程素养。

### 1.2 定向路由推送与高可靠通讯 (Targeted Push) - [已完成实施]
- **实现方案**：从“网关广播”演进为“精准路由推送 (Lookup-then-Push)”。
- **关键逻辑**：
  - **Message-RPC** 消费 Kafka 后查 **pkg/router** (Redis) 定位用户所在网关。
  - 通过 **internal/push** 接口发起定向推送，极大降低多节点环境下的冗余消耗。
  - 利用 Kafka **Partition Key (ConversationId)** 确保分布式环境下的消息局部时序。
  - 实现 **WebSocket 心跳续约** 机制，确保 Redis 路由记录的实时性。

### 1.3 网关多节点集群化 (Multi-node Gateway Cluster) - [待执行]
- **目标**：消除单点故障（SPOF），实现网关层的水平扩展与故障自愈。
- **实施计划**：
  - **负载均衡**：前端接入层引入 Nginx 或 LVS 实现多网关节点的流量分发。
  - **分布式寻址**：利用已实现的 **pkg/router**，确保无论用户连在哪个网关节点，消息都能通过分布式路由表精准下发。
  - **高可用验证**：模拟单节点宕机场景，验证客户端自动重连及路由记录自动更新的鲁棒性。

## 2. 深度演进预案 (贷款内容)

### 2.1 动态寻址与不停机发布 (Dispatcher)
- **设计**：在网关层实现一个通用的 `DispatchHandler`，通过 Etcd 的 `Watch` 机制动态监听路由映射字典。
- **效果**：新增服务只需在 Etcd 侧更新配置，网关即可瞬间具备转发能力，解决路由变更需重启网关导致的连接中断问题。

### 2.2 极致缓存治理 (Cache 3.0)
- **Singleflight 防击穿**：确保在热点 Key 失效瞬时，只有一个请求打向数据库，其他并发请求阻塞复用结果。
- **缓存降级策略**：当检测到 Redis 集群异常时，系统自动切换为数据库直连（带频率保护），优先保障核心业务不中断。

## 3. 常见面试对线 Q&A 总结

- **问：如何保证系统高可用？**
  - **答**：网关多节点部署消除单点故障；Etcd 租约自动失效 + P2C 算法的主动探测避障实现故障自动转移。
- **问：10万并发怎么支撑？**
  - **答**：网关无状态水平扩展；Kafka 异步削峰保护后端存储；多级缓存（L1 内存 + L2 Redis）降低 I/O 压力。
- **问：为什么选 Protobuf 而不是 JSON？**
  - **答**：强类型契约、规避 JS 大数精度丢失、二进制体积更小、解析速度快 5-10 倍。
- **问：如何处理推送风暴？**
  - **答**：采用“发收分离”架构与定向路由下发。Message-RPC 查路由后只对特定网关节点发起推送请求，避免全量广播带来的资源浪费。
