## 2. Protoc 至 Buf CLI 迁移方案

### 2.1 背景与动机
本项目目前依赖传统的 `protoc` 编译器和复杂的 `Makefile` 脚本进行代码生成。随着微服务数量增加，面临以下问题：
- **插件管理困难**：不同环境下 `protoc-gen-go` 版本不一致导致生成代码差异。
- **配置冗长**：`protoc` 命令行参数极其复杂，难以维护。
- **缺乏标准**：缺少自动化的 Lint 检查和 Breaking Change 检修，容易引入不兼容的 API 变更。

### 2.2 目标架构
迁移后，系统将使用 **Buf CLI** 统一管理所有 Protobuf 定义。Buf 提供了一套完整的工具链，包括编译、生成、Lint、格式化和版本管理。

### 2.3 详细迁移步骤

#### 阶段一：配置文件集成
1. **初始化模块**：在 `rpc/` 目录下创建 `buf.yaml`，定义项目根路径及依赖。
2. **定义生成策略**：创建 `buf.gen.yaml`，配置 `protoc-gen-go` 和 `protoc-gen-go-grpc` 插件及其输出路径。

#### 阶段二：代码生成重构
1. **替换生成逻辑**：删除 `Makefile` 中碎片化的 `protoc` 命令。
2. **一键生成**：统一使用 `buf generate` 扫描并更新所有 `pb.go` 文件。

#### 阶段三：质量保障
1. **Lint 规则应用**：通过 `buf lint` 强制执行命名规范。
2. **兼容性检查**：在 CI/CD 中加入 `buf breaking`，防止破坏性变更。

### 2.4 验证计划
1. **编译一致性**：验证 `buf` 生成的代码是否能与现有的 `goctl` 业务逻辑完美契合。
2. **流程简化度**：评估从修改 `.proto` 到代码生效的步骤是否大幅减少。
